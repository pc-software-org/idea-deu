### 3. Task 10: Alle 72.513 Einheiten übersetzen

Das ist die große, resumierbare Langzeitaufgabe. Immer nur einen Batch bearbeiten und nach erfolgreichem Import committen/pushen.

Nächsten Batch exportieren:

```bash

```

Der aktuelle Batchpfad steht anschließend in `translations/checkpoint.json`. Nur die `target`-Felder des exportierten Batches übersetzen. IDs, Quelltexte, Hashes, Kontext und Batchmetadaten nicht verändern.

Übersetzungsregeln:

- neutral formulieren; „Sie“ nur als Rückfall
- Git, Commit, Branch, Debugger, Breakpoint, API, URL und IntelliJ IDEA beibehalten
- Platzhalter, MessageFormat, printf, Templatevariablen, HTML/XML, Links und Mnemonics exakt bewahren
- keine pauschale, kontextfreie Übersetzung mehrdeutiger Begriffe
- kurze natürliche UI-Texte, keine steifen Wort-für-Wort-Übertragungen

Batch importieren:

```bash
BATCH=$(python3 -c 'import json; print(json.load(open("translations/checkpoint.json"))["current_batch"])')
python3 -m scripts.idea_deu --root . import-batch "$BATCH"
```

Erwartung: keine blockierenden Findings. Falls vorhanden, denselben Batch korrigieren und erneut importieren.

Nach jedem erfolgreichen Batch:

```bash

git add translations glossary reports

git commit -m "i18n: translate IntelliJ batch $SEQUENCE"
git push origin feature/language-pack-pipeline
```

Das wiederholen, bis `OPEN == 0` und `blocking == 0`.

Wichtig: Nicht versuchen, `translations/units.jsonl` frei oder massenhaft außerhalb des Batchprotokolls umzuschreiben. Die Workflow-Sperren, Digests und WAL-Recovery sollen Manipulationen und Teilzustände verhindern.

## Sicherheits-/Architekturhinweise

- Batch-, Scan- und Reportzustände verwenden workflowweite Prozesssperren und crash-recoverbare Transaktionen.
- Kritische Pfade sind über Directory-FDs und `O_NOFOLLOW` verankert.
- Das Quell-ZIP wird einmal über einen gehaltenen FD validiert und gescannt; kein Pfad-Reopen zwischen Hash und Extraktion.
- Generierte Ressourcen und das Paket werden gegen unabhängige Inputs neu verifiziert.
- Der ZIP-Verifier prüft äußere und innere ZIP-Struktur, lokale Header, Central Directory, Flags, Metadaten, Deflate-Level und Bytegrenzen strikt.
- `status` ist read-only und darf Recoverybedarf nur melden, nicht selbst Zustände verändern.
- Keine Scanner-/Validatorgrenze zur Behebung einzelner realer Dateien pauschal lockern. Immer erst eine kleine Regressionfixture schreiben.
